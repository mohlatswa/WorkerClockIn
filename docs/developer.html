WorkClock — Technical Documentation

Architecture, database schema, authentication flows, security model, and function reference for the WorkClock Staff Attendance PWA.

System Overview & Architecture

WorkClock is a Progressive Web App that enables staff to clock in and out using GPS-verified location, with three supported authentication methods: PIN, WebAuthn biometrics (fingerprint / Face ID), and server-side facial recognition via face-api.js.

The application is entirely front-end: a single HTML page (index.html) with no build step, no bundler, and no server-side runtime. All data is persisted in a Supabase (PostgreSQL) hosted database via the Supabase JavaScript SDK (anon key, client-side).

Multi-tenancy is achieved through a companies table. Each company receives a short URL code; workers open the portal via ?c=COMPANYCODE. Admins are scoped to a single company (except for the developer super-role which spans all companies).

Architecture Summary

  • Front-end: Vanilla HTML5 / CSS3 / ES5-compatible JavaScript (no transpiler)
  • State: Single global object S (no framework, no reactive system)
  • Routing: Single-page with visible panels toggled via showPg(id)
  • Backend: Supabase — PostgreSQL database, auto-generated REST API (PostgREST), anon key access
  • Hosting: GitHub Pages (static files, main branch)
  • Auth: Custom (PIN stored as plain text in DB, WebAuthn credentials stored as base64 rawId, face descriptors as JSON float arrays)
  • Security boundary: Device ID + session token stored in localStorage; server-side lockout enforced via locked_until column
⚠
Note on PIN storage: PINs are currently stored as plain text in the workers table. For production hardening, consider hashing PINs server-side using a Supabase Edge Function or PostgreSQL trigger with pgcrypto.

Key File Descriptions

index.html

Contains every page as a <div class="pg"> element. The active page has the active class added by showPg(id). Pages include: home, wlogin, auth, worker, face-scan, admin-login, admin, developer.

app.js

All application logic in a single file. Uses a global state object S. Functions are globally scoped so they can be called from onclick attributes in the HTML. Supabase queries use the db client (initialised in config.js). All async operations are wrapped with withTimeout(promise, ms) to prevent indefinite hangs.

config.js

Loaded before app.js. Creates the global db Supabase client. Also declares EmailJS constants for the optional admin password-reset OTP feature.

sw.js

A self-destructing service worker: on activate it deletes all caches, unregisters itself, and refreshes all open tabs. This is intentional — the app no longer relies on a service worker cache for offline support.

Database Schema

The Supabase PostgreSQL database contains five tables. Row Level Security (RLS) is currently disabled; all access goes through the anon key. For production, RLS policies should be added.

Admin Roles

  • developer: Full system access. Manages all companies, all admin accounts, all workers. Sees the Developer panel on login.
  • super_admin: Manages their own company only. Can add/edit workers, view attendance, create admin and regular admin accounts within the company.
  • admin: Manages their own company workers and attendance. Cannot create other admin accounts. The Admins tab is hidden.

Authentication Flow — Worker PIN

Worker enters Employee ID on the home screen and taps Continue → findWorker() queries workers by employee_id (filtered by company_id if URL has ?c=).

If the account exists and is active, the auth screen is shown with the worker's name and avatar. If biometric is enrolled, the fingerprint button also appears.

Worker taps PIN digits on the numpad (4–6 digits). After 4+ digits, verifyPin() is called automatically after 600 ms.

verifyPin() checks locked_until. If the account is locked and the timestamp is in the future, the attempt is rejected with a countdown message.

The entered PIN is compared to S.worker.pin (loaded from DB). On mismatch, failed_attempts is incremented in the DB. After 5 failures, locked_until is set to NOW() + 30 minutes and failed_attempts resets to 0.

On match, secureSignIn('pin') is called. Device binding is checked; if workers.device_id is set to a different device ID, the sign-in is rejected.

A new 64-char hex session_token is generated, written to localStorage and to the workers row. If no device_id is set, the current device ID is registered. failed_attempts and locked_until are cleared.

enterWorkerDashboard() is called — worker is now signed in.

Authentication Flow — Worker Biometric (WebAuthn)

Worker taps Use Fingerprint / Face ID button on the auth screen (visible only if biometric_enabled = true and window.PublicKeyCredential exists).

authBiometric() calls navigator.credentials.get() with allowCredentials containing the worker's stored biometric_credential_id (decoded from base64).

The platform authenticator (device fingerprint sensor, Face ID, Windows Hello) prompts the user and returns a signed assertion.

On success (no server-side signature verification — the credential presence itself is the proof), secureSignIn('biometric') is called and the normal device-binding + session-token flow completes.

⚠
WebAuthn assertions are not cryptographically verified server-side in this implementation. The credential ID match is used as a trust signal. For higher security environments, add server-side assertion verification via a Supabase Edge Function.

Authentication Flow — Face Recognition

Worker taps Use Face Recognition from the home screen. openFaceRecog() loads the face-api.js library from jsDelivr CDN (if not already loaded), then loads three neural net models: TinyFaceDetector, FaceLandmark68TinyNet, FaceRecognitionNet.

All workers in the company with a non-null face_descriptor are fetched. Their stored Float32Array(128) descriptors are decoded from JSON and wrapped as faceapi.LabeledFaceDescriptors keyed by worker UUID.

A FaceMatcher is created with a distance threshold of 0.50. Distances below this threshold are considered a match (lower = more strict).

The front camera is opened. Each animation frame, faceFrame() runs faceapi.detectSingleFace(video, TinyFaceDetectorOptions).withFaceLandmarks(true).withFaceDescriptor().

The detected descriptor is compared against enrolled descriptors. If the best match distance is ≤ 0.50, the worker is identified. The camera stops, and secureSignIn('face') is called after a 900 ms confirmation delay.

Authentication Flow — Admin Login

Admin enters username and password on the Admin Login page and clicks Login → adminLogin().

First, a lockout pre-check: the account row is fetched by username to read is_active, failed_attempts, and locked_until. If locked, the attempt is rejected immediately without a DB password comparison.

Full query: admin_users WHERE username = ? AND password_hash = ? AND is_active = true.

On failure, failed_attempts is incremented. At 5 failures, locked_until = NOW() + 30 min and failed_attempts resets to 0.

On success, failed_attempts and locked_until are cleared. S.admin is set. Role-based routing: developer goes to pg-developer, others go to pg-admin. The Admins tab is hidden for admin role.

Security — Device Binding

When a worker first signs in successfully, a UUID (generated via crypto.randomUUID() or a polyfill) is stored in localStorage under wc_device_id and written to workers.device_id in the database.

On every subsequent sign-in, secureSignIn() compares the current device's localStorage UUID with the stored device_id. If they differ, sign-in is rejected:

if (w.device_id && w.device_id !== deviceId) {
  showErr('err-pin', '🔒 This account is bound to another device.');
  setTimeout(function() { S.worker = null; showPg('wlogin'); }, 3000);
  return;
}

An admin can reset device binding using the 🏝 Reset Security button (shield icon) in the Workers tab. This sets device_id, session_token, failed_attempts, and locked_until to null/0.

Security — Session Tokens

On every successful sign-in (PIN, biometric, or face), a new 64-character cryptographically random hex token is generated:

function genToken() {
  var arr = crypto.getRandomValues(new Uint8Array(32));
  return Array.from(arr).map(function(b) { return b.toString(16).padStart(2, '0'); }).join('');
}

The token is stored in both localStorage (wc_session_token) and workers.session_token. On sign-out, the DB column is set to NULL and the localStorage entry is removed. This invalidates any other active sessions for that worker immediately.

Security — Account Lockout Algorithm

The same algorithm is used for both workers (PIN) and admins (password):

  • Each failed attempt increments failed_attempts in the DB
  • At exactly 5 failures: locked_until = NOW() + 30 minutes, failed_attempts resets to 0
  • While locked_until is in the future, all attempts are rejected immediately
  • On successful authentication: failed_attempts = 0, locked_until = null
  • Admin can reset worker lockout via the security reset button (clears both failed_attempts and locked_until)
🔒
The lockout check for admins is done in two stages: a pre-check reads lockout state before the password comparison, so the lockout message is shown immediately without revealing whether the username exists.

Clock-in Methods

1 — Employee ID + PIN

The default method. Worker enters their employee_id, then taps a 4–6 digit PIN on a custom numpad. The PIN is auto-submitted 600 ms after the 4th digit. The auth_method value written to attendance is 'pin'.

2 — Fingerprint / Face ID (WebAuthn)

Uses the W3C WebAuthn platform authenticator API. Credentials are registered with authenticatorAttachment: 'platform', residentKey: 'required', and userVerification: 'required'. The credential's rawId is base64-encoded and stored in workers.biometric_credential_id. The worker's UUID is the WebAuthn user ID. The auth_method value is 'biometric'.

3 — Facial Recognition (face-api.js)

Client-side only. face-api.js v0.22.2 runs in the browser using TensorFlow.js under the hood. The TinyFaceDetector model (fast, lightweight) is used for detection. A 128-float embedding vector is generated from the face. During recognition, cosine distance against stored embeddings is computed. Threshold is 0.50. The auth_method value is 'face'.

Geolocation Enforcement

Regardless of authentication method, the Clock In/Out button is only enabled when the worker's GPS coordinates are within workplaces.radius_meters of the workplace centre. Distance is calculated using the Haversine formula in haversineM(). The location is polled every ~15 seconds via navigator.geolocation.watchPosition().

Geolocation Detail

function haversineM(la1, lo1, la2, lo2) {
  var R = 6371000; // Earth radius in metres
  var r = function(d) { return d * Math.PI / 180; };
  var a = Math.pow(Math.sin(r(la2-la1)/2),2)
        + Math.cos(r(la1)) * Math.cos(r(la2))
        * Math.pow(Math.sin(r(lo2-lo1)/2),2);
  return R * 2 * Math.atan2(Math.sqrt(a), Math.sqrt(1-a));
}

The GPS coordinates at clock-in and clock-out are stored in the attendance record. If a worker's workplace is not configured (workplace_id = NULL), a warning is shown but clock-in is still permitted.

Face Recognition Detail

Models are loaded lazily from jsDelivr CDN (https://cdn.jsdelivr.net/gh/justadudewhohacks/face-api.js@0.22.2/weights) and cached in memory for the session. Three models are required:

  • tinyFaceDetector — Fast face detection (input 320px, score threshold 0.5)
  • faceLandmark68TinyNet — 68-point facial landmark detection
  • faceRecognitionNet — 128-float embedding generation

Enrolment: The admin (or worker) captures a single image frame, runs detection+embedding, and stores the JSON-serialised Float32Array in workers.face_descriptor.

Recognition: All active workers' descriptors are loaded and used to build a FaceMatcher. Recognition runs per-animation-frame until a match is found or the user cancels.

PWA Details

WorkClock is installable as a Progressive Web App via the browser's "Add to Home Screen" prompt. The manifest declares:

  • display: "standalone" — no browser chrome when launched from home screen
  • orientation: "portrait-primary" — locked to portrait
  • theme_color: "#2563EB" — blue status bar on Android
  • start_url: "./index.html"
  • Shortcut: Clock In / Out → ./index.html#home

The service worker (sw.js) is self-destructing: it deletes all caches and unregisters itself on activation. The app does not cache resources for offline use — an internet connection is required to reach Supabase.

Adding a New Company (Step by Step)

  1. Log in to the admin panel as a developer account.
  2. On the Developer panel, locate the Companies section and click + Add Company.
  3. Enter the company name and a short unique code (e.g. REATLEGILE, uppercase, no spaces).
  4. The company is created with is_active = true.
  5. Create a super_admin account for the company: switch to the Admin Users section, click + Add Admin, set role to super_admin, and associate with the new company.
  6. Log in as that super_admin. Go to the Setup tab and copy the Worker Portal Link:
    https://mohlatswa.github.io/WorkerClockIn/?c=YOURCODE
  7. Add at least one Workplace with GPS coordinates and a radius. Workers assigned to this workplace must be within the radius to clock in.
  8. Add workers via the Workers tab. Distribute the portal link.

Extending the System

Adding a new page

Add a <div id="pg-mypage" class="pg"> inside index.html. Navigate to it with showPg('mypage'). All existing pages follow this pattern.

Adding a new clock-in method

  1. Add a new button/flow in index.html on the home page or auth page.
  2. On success, call secureSignIn('mymethod') after setting S.worker.
  3. Add 'mymethod' to the auth_method CHECK constraint in the database.

Enabling RLS

Currently RLS is disabled. To enable it, you need to create policies for each table. The anon key should only be able to SELECT/INSERT/UPDATE rows relevant to the requesting session. Consider using Supabase Auth or custom JWT claims for this.

Hashing PINs

Add a PostgreSQL trigger using pgcrypto:

CREATE EXTENSION IF NOT EXISTS pgcrypto;

CREATE OR REPLACE FUNCTION hash_pin() RETURNS TRIGGER AS $$
BEGIN
  IF NEW.pin IS DISTINCT FROM OLD.pin THEN
    NEW.pin := crypt(NEW.pin, gen_salt('bf'));
  END IF;
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_hash_pin
BEFORE INSERT OR UPDATE ON workers
FOR EACH ROW EXECUTE FUNCTION hash_pin();

Then update the PIN verification in verifyPin() to use a DB function call instead of a client-side string comparison.
